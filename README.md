# 智能食堂预订系统

基于 Flask + SQLite + 原生前端的食堂预订系统，支持手机 / 电脑双端，黑白极简主题。

这是一个练手的全栈项目，把"前端页面 + 后端接口 + 数据库"完整跑通，并做了管理端做数据看板。

## 技术栈

- 后端：Python 3.8+ / Flask 3.1 / Flask-CORS
- 数据库：SQLite（WAL 模式）
- 前端：HTML5 + CSS3 + 原生 JavaScript（响应式，支持手机 / 电脑）

## 功能

### 用户端（手机 / 电脑自适应）
- 登录（姓名 + 工号）
- 查看今日菜单（卡片式，含营养信息、过敏标签）
- 菜品搜索（菜名 + 别名模糊匹配）
- 在线下单（自动校验截止时间、库存、限购）
- 今日 / 历史订单查询
- 健康饮食建议
- 取餐提醒

### 管理端（手机 / 电脑自适应）
- 管理员首次绑定（进入时绑定姓名 + 工号）
- 数据概览：今日订单 / 待取餐 / 已取餐统计卡片、最近订单、库存概览（进度条）、热销 TOP5、低库存预警
- 菜品管理（上架 / 下架 / 改库存 / 补货 / 改限购）
- 订单管理（核销 / 取消 / 列表查询）
- 系统配置（下单截止时间、取餐时段）
- 操作日志、超时订单处理

## 我的工作

- 设计三层结构（`database` / `server` / `client`）
- 实现下单校验逻辑（截止时间、库存、限购）
- 做管理端数据看板与交互
- 写一键启动脚本（Windows / Linux / Mac）

## 运行方式

```bash
pip install flask flask-cors
python database/init_db.py
python server/app.py
```

服务启动后：用户端 http://localhost:5000 ，管理端 http://localhost:5000/admin 。也可直接用 `start.bat` / `start.sh`。

## 项目不足

- 用户认证很轻（姓名 + 工号），不是真账号体系
- 没有做高并发优化
- 管理端权限简单，适合自用演示

## 后续计划

- 换成更严谨的认证方式
- 补接口测试
- 视需要加容器化部署

## Reflection

这是我第一个结构比较完整的全栈项目。最大的收获是把"业务规则"（比如下单约束）落到代码里的思维方式——需求里一句"不能超库存"，翻译成代码就是好几处校验。
